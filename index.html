b. Reflexión — Aprendizaje y EstándaresReflection — Part 2 (b. Reflexión — DAS)
¿Qué aprendimos? Una línea base de arquitectura de software es fundamental para garantizar la calidad, mantenibilidad y escalabilidad de un sistema. Sin una arquitectura definida, los procesos de desarrollo se vuelven lentos, los costos de mantenimiento aumentan y la calidad del producto final se ve comprometida. Este proyecto nos permitió experimentar de primera mano el ciclo de vida del diseño arquitectónico.
Estándares aplicados: Aprendimos a documentar siguiendo la norma ISO/IEC/IEEE 42010:2022 para descripción de arquitecturas, ISO 12207 para procesos del ciclo de vida del software, y ISO 27002 para controles de seguridad. Cada norma aportó una dimensión distinta: la primera nos dio el marco conceptual, la segunda la estructura de procesos, y la tercera las guías de protección de datos.
b. Reflexión — Habilidades y DesafíosReflection — Part 3 (b. Reflexión — DAS)
Habilidades desarrolladas: Modelado UML (casos de uso, clases, secuencia, paquetes, componentes), diseño de bases de datos relacionales, aplicación de patrones como MVC, Repository, Dependency Injection, principios SOLID, y documentación técnica profesional. Trabajamos como equipo multidisciplinario usando Git para control de versiones.
Desafíos superados: La coordinación entre miembros del equipo, la interpretación correcta de los requisitos de la farmacia, y la selección del stack tecnológico adecuado fueron retos que enfrentamos y resolvemos mediante reuniones semanales y revisiones cruzadas de código y documentación.

b. Reflection — Learning & StandardsReflection — Part 2 (b. Reflection — DAS)
What did we learn? A software architecture baseline is essential to guarantee quality, maintainability, and scalability. Without a defined architecture, development processes become slow, maintenance costs increase, and final product quality is compromised. This project allowed us to experience the architectural design lifecycle first-hand.
Applied standards: We learned to document following ISO/IEC/IEEE 42010:2022 for architecture description, ISO 12207 for software life cycle processes, and ISO 27002 for security controls. Each standard contributed a different dimension: the first provided the conceptual framework, the second the process structure, and the third the data protection guidelines.

We worked as a multidisciplinary team using Git for version control.
Overcome challenges: Team coordination, correct interpretation of pharmacy requirements, and selection of the appropriate technology stack were challenges we faced and resolved through weekly meetings and cross reviews of code and documentation.

d. Situación Problemática — IntroducciónProblem Statement — Part 1
La empresa The Repair of System no cuenta con una línea base de arquitectura de software definida, provocando procesos de desarrollo lentos, baja calidad y altos costos de mantenimiento. La Farmacia Santa Cruz carece de un sistema automatizado de compra y venta, dificultando el control de inventario y fechas de vencimiento.

El Núcleo del Problema

La inexistencia de una estructura formal impide la estabilidad técnica y ralentiza las operaciones comerciales de la farmacia, generando pérdidas económicas directas y desorganización general.

d. Situación Problemática — DesgloseProblem Statement — Part 2

Problemas Técnicos

Procesos sin metodología, ausencia de documentación arquitectónica, código sin patrones estructurados y gran dificultad para incorporar nuevos desarrolladores.

Problemas Operativos

Control de inventario manual en cuadernos, desconocimiento de caducidades cercanas, pérdidas de ventas por rotura de stock y atraso en pedidos.

Riesgos

Incumplimiento de regulaciones sanitarias (MINSAL), errores humanos graves de registro, pérdida total de datos por falta de respaldos confiables.

d. Problem Situation — IntroductionProblem Statement — Part 1
The company The Repair of System does not have a defined software architecture baseline, causing slow development processes, low quality, and high maintenance costs. Farmacia Santa Cruz lacks an automated purchase and sale system, making inventory control and expiration date management difficult.

The Root Problem

The lack of a structured architectural base compromises scalability and leaves the business exposed to critical data loss and manual auditing delays.

d. Problem Situation — DetailsProblem Statement — Part 2

Technical Problems

Development processes without defined methodology, lack of structural documentation, code with no standards and high onboarding friction.

Operational Problems

Manual inventory control in notebooks, no visibility of soon-to-expire batches, lost sales due to stockouts and delays in orders.

Risks

Non-compliance with local health regulations (MINSAL), high human error margins in records, and critical loss of data due to lack of backups.

Reglas de Negocio

Venta estricta con regla FIFO por lote, sistema de alertas automatizado para vencimiento (30/15/7 días) y control de stock mínimo.

Flujos Operacionales

Entrada de productos con asociación de número de lote y fecha de vencimiento, validaciones de integridad y auditoría histórica.

e. Proposed Solution — OverviewProposed Solution — Overview
We designed the architecture baseline for the Inventory Management System for Farmacia Santa Cruz (SGIFSC) using Google Cloud Platform as the main infrastructure, with an N-Tier (4-layer) architecture documented under ISO/IEC/IEEE 42010:2022, 12207, and 27002.

Selected Tech Stack

.NET Core 8Angular/ReactMySQLGoogle CloudDockerJWTBootstrap 5Redis

Security & Standards

JWT authentication with Owner and Employee roles, HTTPS communication, Data Protection, 30-minute session expiration, and full Audit Logging.

e. Proposed Solution — Core ComponentsProposed Solution — Part 1

Frontend (UI)

Angular / React + Bootstrap 5. Modern responsive interface featuring dashboards, stock forms, and a secure checkout POS terminal.

Backend Core

.NET Core 8, C#. RESTful API with dependency-injected services, controllers, token verification pipelines, and decoupled handlers.

Database Model

MySQL on Cloud SQL. Relational model mapping Products, Batches, Sales, Suppliers, Categories, Alerts, and Audit Logs.

e. Proposed Solution — Infrastructure & RulesProposed Solution — Part 2

Cloud Infrastructure

Google Cloud Platform. Cloud SQL (Managed MySQL), serverless containers via Cloud Run, secured storage buckets, and Redis Memorystore.

Business Logic

Strict FIFO batch dispatching, automated cron-jobs to flag critical batches at 30/15/7 days, and low stock threshold alarms.

Operational Flows

Efficient drug intake tracking with immediate batch assignment, automated stock subtraction, state transitions, and granular activity logs.

f. Síntesis del DAS — Datos, Seguridad y ReglasDAS Synthesis — Part 2

04 Modelo y Motor

Normalización estricta de MySQL 8.0 en la nube con particionamiento de lotes y almacenamiento de archivos planos.

05 Mecanismos de Seguridad

Tokens JWT con caducidad, encriptación SHA256, canal HTTPS forzado y logs de auditoría inmutables.

06 Lógica FIFO y Alertas

Despacho automatizado priorizando el vencimiento más cercano, evitando mermas y enviando notificaciones proactivas.

f. Síntesis del DAS — Flujos OperacionalesDAS Synthesis — Part 3

07 Registro de Productos y Lotes

Flujo seguro para persistir datos del producto (categoría, stock mínimo, costo). Se encadena con el registro de lotes que capturan el número de lote, cantidad inicial y fecha de vencimiento precisa.

08 Procesamiento de Ventas (FIFO)

El POS consulta los lotes activos ordenados por vencimiento ascendente. Se deducen existencias secuencialmente del lote más antiguo, manteniendo el control exacto de remanentes.

f. Síntesis del DAS — Trabajos e InfraestructuraDAS Synthesis — Part 4

09 Alertas Programadas

Background worker diario que escanea el estado de los lotes activos, clasificando riesgos en niveles de severidad (vencimiento cercano, retiro o stock crítico).

10 Distribución Cloud

Hospedaje en Google Cloud mediante Cloud Run para elasticidad, Cloud SQL de alta disponibilidad y caché compartida por Redis.

f. Detalle de Capas — Presentación y NegocioLayers Detail — Part 1

Capa de Presentación (UI)

Responsabilidad: Renderizado de interfaces limpias, captura del flujo de entrada del usuario y presentación interactiva de datos procesados.

En SGIFSC: Formularios para transacciones, buscador de medicamentos en el POS y gráficos para la propietaria.

Herramientas: Angular 17/React 18, Bootstrap 5, Font Awesome, Chart.js.

Capa de Lógica de Negocio (BLL)

Responsabilidad: Validación de reglas comerciales, orquestación de tareas complejas e intermediario lógico inmutable.

En SGIFSC: InventarioService para stock, VentaService con estrategia FIFO y AlertaService para mermas.

Herramientas: .NET Core 8, C#, AutoMapper, FluentValidation.

f. Detalle de Capas — Datos y AlmacenamientoLayers Detail — Part 2

Capa de Acceso a Datos (DAL)

Responsabilidad: Abstracción total de la infraestructura física de persistencia, facilitando operaciones CRUD sin código SQL plano.

En SGIFSC: Repositorios genéricos (ProductoRepository, LoteRepository, VentaRepository) bajo Unit of Work.

Herramientas: Entity Framework Core, MySQL Connector, LINQ.

Base de Datos

Responsabilidad: Almacenamiento consistente, seguro y veloz de todas las entidades, asegurando la integridad referencial.

En SGIFSC: MySQL optimizado con índices en campos clave y Stored Procedures para operaciones recurrentes.

Herramientas: MySQL 8.0 en Google Cloud SQL.
